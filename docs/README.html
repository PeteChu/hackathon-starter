Final readiness checklist

Before the hackathon submission freeze, confirm each of these. Your checks are saved in this browser.

0 / 11 ready ✓ Submission ready
  • problem-brief.md — user, pain, success metric, MVP scope, and wow moment are all filled
  • architecture.md — components, request flow, data model, config, production path are described
  • api-contract.md — endpoint table is populated with real demo-slice endpoints only
  • task-board.md — Now tasks are complete, Cut list is explicit
  • test-plan.md — critical journey is tested, smoke check passes, demo risks are documented
  • demo-script.md — hook, live flow (step by step), fallback, and Q&A answers are written
  • pitch.md — one-liner, demo proof, impact, credibility, and ask are filled
  • judging-scorecard.md — scores and judge-visible evidence are filled for each criterion
  • hackathon-ai-workflow-plan.html — kept as reference (no changes needed unless the workflow changes)
  • All docs use consistent terminology with CONTEXT.md (if present)
  • No stale starter-only references remain (e.g. /api/ideas references, idea-board screenshots)

Traceability map — problem → proof → judging

Each element of the problem brief should trace through architecture, API, tasks, tests, and demo to the judging scorecard. Use this map to verify end-to-end traceability once the challenge is known.

problem → proof → judging

How to use

  1. After the challenge is known, fill each doc using the skills in the recommended order.
  2. Trace forward: pick a row in the Problem Brief column and verify the corresponding columns in Architecture, Tasks, Test, and Demo are covered.
  3. Trace backward: pick a scorecard criterion and verify there is evidence (or at least a plan) in the upstream docs.
  4. If a link is missing, either the doc needs an update or the scope needs an explicit cut decision in task-board.md.
