Phase 0 · before invoking skills

Preflight the repo and define the source of truth.

1Confirm runtime

Use Node 20+ and pnpm. Install dependencies if needed, then run the starter checks. 

make install
make test
make build

2Run the starter

Open two terminals and verify the placeholder demo works before replacing it. 

make dev-node   # API on :8080
make dev-web    # UI on :3000
make smoke      # with API running

3Lock the rule

One working vertical slice beats many partial features. Log assumptions in docs/problem-brief.md unless they involve secrets, safety, or external access.

If the starter or skills are missing in another clone: ask for the missing path before applying this plan. In this workspace, both the starter files and .claude/skills are present, so no clarification is required.

Action plan · setup to completion

Step-by-step developer guideline

0. Setup gate

Baseline the starter before changing it

No skill yet

Run the starter as-is so later failures are attributable to your changes, not the template. 

make install
make test
make build
make dev-node   # terminal A
make dev-web    # terminal B
make smoke      # terminal C, API must be running
  • If install/test/build fails, fix only the blocker required to run the starter.
  • Do not add product scope until /problem-framer captures the official challenge.
1. Frame

Use /problem-framer to choose the winning problem

Docs first

When: immediately after receiving the challenge, or any time the team is debating “what are we actually building?”

How: provide the official challenge text, time budget, known constraints, and ask it to use docs/judging-scorecard.md. 

/problem-framer "Challenge: <paste official challenge>. Time budget: <hours>. Team: <people>. Use docs/judging-scorecard.md and protect a single end-to-end MVP slice."
  • Confirm docs/problem-brief.md has no empty headers.
  • Demand a demo proof metric that can appear on screen in seconds.
  • Record assumptions instead of blocking, except for safety, secrets, or external-system access.
  • Optionally run /grill-with-docs to stress-test the brief against the domain model and sharpen terminology.
2. Design

Use /solution-architect to pin the fastest production-shaped architecture

No code yet

When: after the problem brief is accepted and before changing routes, storage, or UI.

How: ask for a design within the starter stack: Fastify 5 + Zod 4 API, Next.js 16 App Router, Tailwind v4, pnpm/turbo. 

/solution-architect "Use docs/problem-brief.md. Design only the MVP vertical slice. Prefer in-memory storage behind a module interface; no new dependency unless it removes real demo risk."
  • Check docs/architecture.md has components, request flow, data model, config/secrets, production path, and tradeoffs.
  • Name the riskiest component to build first; this informs the first task.
  • For frontend work, remember apps/web/CLAUDE.md: read local Next.js 16 docs under node_modules/next/dist/docs/ when unsure.
  • Optionally run /grill-with-docs to validate the architecture against the brief and surface hidden assumptions.
3. Contract

Use /api-contract to remove backend/frontend guesswork

Demo wiring risk

When: after architecture, before backend and frontend work start in parallel.

How: give the main entity or feature name. Keep the contract minimal: every response field must be rendered by the UI. 

/api-contract "<feature/entity name> for the first MVP slice. Include only endpoints needed for the live demo, JSON error shape, Zod rules, examples, and frontend fetch notes."
  • Ensure GET /healthz remains documented.
  • Replace or remove starter-only endpoint notes that do not exist in current apps/api/src/server.ts.
  • List exactly three tests for /test-fast: happy path, validation error, and empty/not-found state.
4. Slice

Use /task-slicer to defend the deadline

Cut list required

When: once the contract exists, and again whenever new ideas threaten the demo path.

How: pass real remaining time, team size, and any judging criteria. Force the first task to be UI → API → proof metric, even if ugly. 

/task-slicer "Remaining time: <hours>. Team size: <people>. Use docs/problem-brief.md, docs/architecture.md, docs/api-contract.md, and docs/judging-scorecard.md. First task must be a complete vertical slice."
  • Every task has outcome, likely files, acceptance check, effort, owner, and cut decision.
  • The first task is runnable via /implement-vertical-slice "<slice name>".
  • Move Should-ship items to Cut the moment the first slice is at risk.
5. Build

Use /implement-vertical-slice for exactly one slice

Manual code edits

When: only after docs are ready and the task board names the first slice. This skill edits code, so keep the argument precise.

How: ask it to touch the smallest file set, starting backend-first, then test, then UI. For this starter, the replaceable demo files are: 

apps/api/src/lib/store.ts
apps/api/src/lib/ideas.ts
apps/api/src/server.ts      # replace /api/ideas routes with real slice routes
apps/api/src/ideas.test.ts  # rename/replace with real route tests
apps/web/app/page.tsx
apps/web/lib/api.ts
.env.example                # mirror any new env var, placeholders only
/implement-vertical-slice "<first slice name from docs/task-board.md>"
  • Backend: route in server.ts, Zod safeParse, JSON error shape, no hardcoded secrets.
  • Test: vitest against buildServer(), covering happy path and validation failure.
  • Frontend: headline, value proposition, primary action, results area, proof metric, plus loading/empty/success/error states. Use /shadcn to add UI components (Button, Card, Input, etc.) instead of writing custom HTML.
  • Verification: run make test, then make build, then click the primary workflow with both dev servers running.
  • Report files changed, actual command results, one remaining risk, and update docs/task-board.md.
6. Verify

Use /test-fast immediately after each slice

Prevent demo embarrassment

When: after implementation, before starting the next feature, and again at final freeze.

How: focus on the critical journey, API validation, typecheck/build, and smoke test. Update scripts/smoke-test.sh if the reference idea routes are replaced. 

/test-fast "<slice name or highest-risk area>. Verify docs/api-contract.md matches the implemented API and update docs/test-plan.md."
  • Run make test first; it should exercise Fastify via buildServer().
  • Run make build to catch TypeScript and Next build issues.
  • Start API with make dev-node, then run make smoke against the current real endpoints.
  • Document untested areas and whether they are true demo risks.
7. Iterate or cut

Repeat only after the first slice is showable

Scope control

If there is time after a verified slice, choose the next highest judge-visible item from docs/task-board.md. If the board is stale, rerun /task-slicer with the remaining time. 

/task-slicer "Remaining time: <hours left>. First slice is working. Re-rank Next/Later/Cut by judge-visible ROI and preserve final presentation time."
  • Prefer polish that makes value obvious over hidden technical expansion.
  • Do not start a feature that cannot pass /test-fast before the demo freeze.
  • Take screenshots or a short screen recording after every working checkpoint.
8. Story

Use /demo-story in the final third

Presentation wins

When: as soon as there is a working path; do not wait until the last minutes.

How: give the demo slot length and ask it to tie the live flow to the proof metric and scorecard. 

/demo-story "Demo length: <minutes>. Use docs/problem-brief.md, docs/architecture.md, docs/judging-scorecard.md, and the verified working slice. Include fallback plan and judge Q&A."
  • docs/demo-script.md opens with a 10-second hook: user, pain, value, proof metric.
  • docs/pitch.md has one-liner, demo proof, company impact, production credibility, and pilot ask.
  • Rehearse against security, scalability, adoption, and data-quality questions.
  • Run /frontend-slides to produce a polished HTML presentation deck from the demo script for the live pitch.
9. Freeze

Stop building and lock the demo path

No broad refactors

In final hardening, fix only the highest-impact blockers. Keep one browser tab, one API terminal, one UI terminal, and one fallback recording ready. 

make test
make build
make dev-node
make dev-web
make smoke
  • Fresh browser load works.
  • Primary action works end to end.
  • Proof metric is visible without explanation.
  • Error/empty states are not embarrassing.
  • Fallback screenshot/recording is available locally.
Starter-specific operating notes

Use the template’s seams instead of refactoring around them.

Backend seam

apps/api/src/server.ts exports buildServer(). Keep it that way so vitest can exercise routes with app.inject() without booting a port.

Domain schemas and in-memory data can live under apps/api/src/lib/. If storage is module-level, export a test-only reset helper and call it in afterEach.

Frontend seam

apps/web/app/page.tsx is currently a client component for the reference idea board. Replace it with a 10-second-understandable workflow: headline, value proposition, primary action, results, and proof metric.

Keep fetch helpers in apps/web/lib/api.ts. Read NEXT_PUBLIC_API_URL and render API error messages in the error state.

Docs seam

The existing docs are intentionally draft placeholders. Treat these files as the handoff contract between skills:

docs/problem-brief.md, docs/architecture.md, docs/api-contract.md, docs/task-board.md, docs/test-plan.md, docs/demo-script.md, and docs/pitch.md.

Skills live under both .agents/skills/ and .claude/skills/.

Verification seam

make test runs API tests and web typecheck when dependencies exist. make build builds API and UI. make smoke curls the running API.

After replacing /api/ideas, update scripts/smoke-test.sh so smoke checks follow the real demo path, not the starter route.

Rules that override enthusiasm

Do not read or expose real secrets. Use .env.example placeholders.
Do not add dependencies unless they remove meaningful risk or save major time.
Do not build hidden infrastructure unless it unblocks the demo or production credibility.
Do not start feature two until feature one is working end to end.
Always end implementation with verification and doc updates.
